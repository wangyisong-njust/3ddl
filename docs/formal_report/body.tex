\chapter{Introduction}\label{introduction}}

\hypertarget{motivation}{%
\section{Motivation}\label{motivation}}

Three-dimensional X-ray inspection is increasingly important for
advanced semiconductor packaging, high-bandwidth memory stacks, and
buried interconnect structures because many critical defects cannot be
observed reliably from surface images alone. In this setting, the
inspection workflow is not limited to classification or segmentation. It
must also support geometry-sensitive metrology, such as bond line
thickness, pad misalignment, solder extrusion, and void-related
measurements. That requirement makes the acceleration problem fundamentally harder
than standard semantic segmentation: \emph{most acceleration studies
optimize model latency or segmentation overlap, but this work optimizes
deployment-safe whole-pipeline behavior under thresholded metrology.}
A faster model is not automatically a better deployment choice if its
outputs change the downstream measurements used by engineers and
inspectors.

This report studies that problem on a practical system built from two
repositories. The \emph{wp5-seg} repository provides 3D segmentation
training, evaluation, pruning, finetuning, ONNX export, TensorRT build
scripts, and deployment-oriented benchmarking. The \emph{intelliscan}
repository performs complete raw-scan inference through 2D multi-view
detection, 3D box generation, 3D segmentation, metrology analysis, GLTF
generation, and PDF reporting. The central challenge addressed in this
report is that optimization opportunities exist at model, algorithm,
runtime, and systems levels, but these opportunities interact. A
model-level gain can be weakened or amplified by pipeline overhead.
Likewise, a runtime shortcut that appears harmless at voxel level can
become unacceptable after metrology thresholds are applied.

\hypertarget{metrology-as-a-cliff-sensitive-output}{%
\subsection{Metrology as a Cliff-Sensitive
Output}\label{metrology-as-a-cliff-sensitive-output}}

The downstream metrology stage is cliff-sensitive rather than smoothly
varying. In the current pipeline, defect logic is triggered after
geometric quantities such as pad shift and pillar width are estimated
from the predicted component masks. A simplified form of the
pad-misalignment decision can be written as

\[
d_{\mathrm{misalign}} = \mathbb{1}\left(\frac{|\Delta_{\mathrm{pad}}|}{w_{\mathrm{pillar}}} > \tau_{\mathrm{pad}}\right),
\qquad \tau_{\mathrm{pad}} = 0.10
\]

where \(\Delta_{\mathrm{pad}}\) is the measured pad offset and
\(w_{\mathrm{pillar}}\) is the local pillar width. The important point
is not the exact notation, but the thresholded structure: if a
segmentation boundary moves by only a few voxels, the normalized ratio
can cross the decision threshold and flip the output defect flag
abruptly. This explains why the present work evaluates acceleration
against metrology, not against Dice alone.

\centering
\includegraphics[width=\textwidth]{../figures/metrology_cliff_effect.png}
\caption{Conceptual cliff effect of thresholded metrology: a small
normalized boundary shift can cross the decision threshold and flip the

\end{figure}

\hypertarget{industrial-deployment-challenge}{%
\subsection{Industrial Deployment
Challenge}\label{industrial-deployment-challenge}}

Industrial semiconductor inspection differs from benchmark segmentation
in two important ways. First, the deployment target is not a single
neural network kernel. It is a full workflow containing data loading, 2D
detection, 3D merge, 3D segmentation, measurement logic, visualization,
and reporting. Second, the end users are not evaluating masks directly.
They care about bump-level measurements, defect flags, and engineering
traceability. This means that runtime optimization must preserve not
only segmentation overlap, but also the geometric and thresholded
outputs that drive inspection decisions {[}9,10{]}.

The practical consequence is that standard acceleration techniques such
as pruning, quantization, TensorRT deployment, and graph compilation
must be re-evaluated after integration. A method that looks strong in
isolated inference benchmarking can still be unsafe when it changes the
class boundaries used by downstream metrology {[}8,11{]}. This
deployment gap is the main reason the present work uses a whole-pipeline
evaluation methodology instead of a model-only ranking.

\hypertarget{difference-from-earlier-acceleration-studies}{%
\subsection{Difference from Earlier Acceleration
Studies}\label{difference-from-earlier-acceleration-studies}}

Many prior studies focus on one layer of the problem at a time: faster
detection, faster segmentation, model compression, or runtime
deployment. The present report differs in emphasis. It treats the
inspection stack as a coupled system and evaluates acceleration
decisions at two levels: segmentation quality on the dataset-level
evaluation set, and downstream stability after whole-pipeline
deployment. This is the main methodological difference between this
report and more conventional ``speed versus Dice'' studies.

\hypertarget{problem-statement}{%
\section{Problem Statement}\label{problem-statement}}

The objective of this report is to accelerate the full 3D semiconductor
inspection workflow while keeping defect-sensitive accuracy acceptable.
Two concrete questions drive the work:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Which model-compression strategies produce a meaningful
  latency-size-quality tradeoff for the 3D segmentation network?
\item
  Which pipeline-level optimizations reduce end-to-end runtime without
  causing hidden drift in segmentation, defect flags, or metrology?
\end{enumerate}

The report therefore does not treat acceleration as a single-stage
benchmark problem. Instead, it studies the complete workflow from raw
input to final engineering outputs.

\hypertarget{platform-under-study}{%
\section{Platform Under Study}\label{platform-under-study}}

\subsection{3D Semiconductor Inspection Domain}

The data studied in this report are 3D X-ray computed-tomography reconstructions of advanced semiconductor packages. Each raw scan is a single-channel volumetric NIfTI file capturing the internal structure of a multi-die stack at 0.7\,$\mu$m isotropic voxel resolution. The structures of interest include solder bumps, copper pillars, copper pads, and inter-layer connections. Defects such as voids (gas pockets trapped in solder) and geometric anomalies (pad misalignment, solder extrusion) are invisible from the package surface and can only be detected through volumetric analysis.

The inspection workflow processes each raw scan through a sequence of stages: (i)~slice extraction from the 3D volume along two orthogonal views, (ii)~2D object detection to locate individual bump regions, (iii)~3D bounding-box generation by merging detections across views, (iv)~per-bump 3D semantic segmentation into five component classes, (v)~geometric metrology computation from the predicted masks, and (vi)~human-readable reporting.

\subsection{Dataset and Segmentation Classes}

The segmentation model assigns each voxel to one of five semantic classes:

\begin{table}[H]
\centering
\caption{Semantic segmentation class definitions}
\label{tab:classes}
\renewcommand{\arraystretch}{1.15}
\begin{tabular}{c l l}
\toprule
\textbf{Class} & \textbf{Label} & \textbf{Description} \\
\midrule
0 & Background    & Air, substrate, and non-structural material \\
1 & Copper Pillar & Vertical interconnect between die layers \\
2 & Solder        & Bonding material between pillar and pad \\
3 & Void          & Gas pocket trapped within solder (primary defect) \\
4 & Copper Pad    & Horizontal landing pad on the memory die \\
\bottomrule
\end{tabular}
\end{table}

Classes are highly imbalanced: background dominates each crop, while void (class~3) is the rarest and most clinically important class. The dataset contains 787~labelled 3D bump crops from 72~distinct scan serial numbers, split into 613~training samples (57~serials) and 174~test samples (15~serials). For whole-pipeline validation, eight raw scans (SN002, SN003, SN008, SN009, SN010, SN011, SN012, SN061) are used, with SN009 (114~detected bumps) serving as the primary benchmark. The eight scans were selected to cover a range of bump counts (95--114), die configurations (single-die and interposer stacks), and both clipped and non-clipped crop geometries.

\subsection{Training Configuration}

The teacher model (MONAI BasicUNet, features $(32, 32, 64, 128, 256, 32)$, 5.75M parameters) was trained with the following hyperparameters: Novograd optimizer, learning rate $1 \times 10^{-3}$, batch size~4, 30~epochs, MultiStepLR schedule (milestones at 60\% and 85\% of training), loss $0.5 \cdot L_\text{CE} + 0.5 \cdot L_\text{Dice}$ with ignore label~6. Input ROI size is $(112, 112, 80)$ voxels. Data augmentation consists of random flips on all three spatial axes and random spatial cropping. Preprocessing applies percentile clipping (p1--p99) followed by z-score normalization. All training uses mixed precision (bfloat16) on NVIDIA L40 GPUs (48\,GB VRAM).

\subsection{Software and Hardware Platform}

The implementation consists of two repositories: \emph{wp5-seg} for model training, pruning, distillation, ONNX export, TensorRT engine building, and deployment benchmarking; and \emph{intelliscan} for end-to-end raw-scan inference including detection, segmentation, metrology, and reporting. The detection stage uses a YOLO-based 2D detector applied to two orthogonal slice views. Energy consumption is measured by polling GPU power draw via \texttt{nvidia-smi} at 200\,ms intervals during pipeline execution and integrating over total runtime.

\hypertarget{main-contributions-of-this-report}{%
\section{Main Contributions of This
Report}\label{main-contributions-of-this-report}}

This report makes four main contributions.

First, it establishes a validated safe engineering acceleration path for
the full \emph{intelliscan} workflow. Combined segmentation and
metrology, report reconstruction without per-bbox raw crop NIfTIs,
removal of per-bbox prediction NIfTIs from the combined default path,
exact quantized in-memory detection, 4-worker CPU image
preparation, and intra-view batch prefetch were all validated on local
raw scans. Together, these changes produce a large end-to-end runtime
reduction without changing downstream outputs on the validated set.

Second, the report builds a first model-compression Pareto front for
\emph{wp5-seg}. Structured pruning ratios 0.375,
0.5, and 0.625 were benchmarked and re-evaluated after
full-train recovery using the same retained knowledge-distillation
recipe. This establishes quality-oriented, balanced, and
aggressive-speed operating points.

Third, the report shows that whole-pipeline deployment is a stricter
gate than full-case Dice. Compressed students with apparently acceptable
full-case recovery can still generate large defect-flag flips and
metrology drift after deployment into \emph{intelliscan}.

Fourth, the report documents several useful negative results. These
include unsafe adaptive-margin behavior driven by context shrinkage,
unsafe larger direct padded ROI behavior, same-GPU cross-view overlap
that remains exact but slows the target stage, and several training-loss
variants that do not survive full-case or whole-pipeline validation.

\hypertarget{evaluation-philosophy-and-the-two-level-deployment-safety-gate}{%
\section{Evaluation Philosophy and the Two-Level Deployment-Safety
Gate}\label{evaluation-philosophy-and-the-two-level-deployment-safety-gate}}

The report adopts a two-level evaluation gate.

At the \emph{wp5-seg} level, a method is considered promising only if
it improves or preserves full-case segmentation quality on the full
evaluation set. Small-slice experiments are used only for rapid
screening and are not used to retain methods.

At the \emph{intelliscan} level, a method is considered acceptable
only if it preserves whole-pipeline behavior on raw scans. This means
that class 3 and class 4 Dice, metrology deltas, and
binary defect-flag flips matter more than background-dominated voxel
agreement alone. This evaluation philosophy is one of the key findings
of the report.

\chapter{Background}\label{background}}

\hypertarget{d-segmentation-and-medical-imaging-style-networks}{%
\section{3D Segmentation and Medical-Imaging Style
Networks}\label{d-segmentation-and-medical-imaging-style-networks}}

U-Net introduced the encoder-decoder skip-connection design that later
became the foundation of many medical and volumetric segmentation
systems {[}17{]}. Three-dimensional encoder-decoder networks such as 3D
U-Net and V-Net extend that idea to volumetric inputs so that local
context can be modeled jointly across all three axes {[}1,18{]}. More
recent self-configuring frameworks such as nnU-Net further showed that
much of segmentation performance comes from carefully matched
preprocessing, training, and inference defaults rather than from
backbone novelty alone {[}19{]}. In highly imbalanced segmentation
tasks, overlap-based losses such as Generalised Dice are often preferred
because minority classes can otherwise be overwhelmed by large
background regions {[}2{]}. Modern healthcare-imaging toolkits such as
MONAI also provide strong reusable implementations for volumetric models
and inference utilities {[}3{]}. Although this report targets
semiconductor inspection rather than medical imaging, the same
architectural and loss-design issues arise because the data are also
volumetric and class-imbalanced.

\hypertarget{model-compression-and-knowledge-distillation}{%
\section{Model Compression and Knowledge
Distillation}\label{model-compression-and-knowledge-distillation}}

Structured pruning removes whole filters or channels and is more
deployment-friendly than unstructured sparsity because it produces
smaller dense models rather than sparse masks {[}4{]}. Channel-level
pruning and slimming methods have been widely used for
deployment-oriented compression because they map naturally to dense
inference libraries and hardware accelerators {[}14,15{]}. Knowledge
distillation transfers information from a larger teacher model to a
smaller student model by matching softened teacher outputs or related
signals {[}5{]}. In segmentation, however, a generic distillation loss
does not always preserve the classes that matter most in downstream
tasks. Boundary-aware and class-aware formulations have therefore been
investigated to better preserve minority structures and class
transitions {[}6{]}. This report uses those ideas as motivation, but
keeps only the variants that actually survive full-case and
whole-pipeline testing.

\hypertarget{detection-and-runtime-deployment}{%
\section{Detection and Runtime Deployment}\label{detection-and-runtime-deployment}}

The \emph{intelliscan} pipeline uses YOLO-style 2D object detection on two orthogonal slice views before 3D bounding-box merge, consistent with the strong speed--accuracy tradeoffs reported in the detection literature {[}12,13{]}. Although the detector is not the final output stage, bounding-box drift propagates directly into crop selection and therefore into segmentation and metrology.

At deployment time, runtime engines such as TensorRT optimize operator fusion, kernel selection, and lower-precision execution {[}7,8{]}, while PyTorch graph compilation (\emph{torch.compile}) can reduce Python overhead through ahead-of-time graph capture {[}11{]}. However, lower latency does not guarantee downstream safety when the application is sensitive to small geometric changes. This report therefore treats both TensorRT and \emph{torch.compile} as potentially valuable but not automatically safe.

\hypertarget{semiconductor-x-ray-inspection-and-metrology}{%
\section{Semiconductor X-ray Inspection and
Metrology}\label{semiconductor-x-ray-inspection-and-metrology}}

Recent work has demonstrated the value of combining detection, segmentation, and metrology in unified 3D X-ray inspection workflows {[}9{]}, while broader literature confirms that internal electronics assessment increasingly relies on volumetric imaging and automated analysis {[}10,20{]}. The present work differs from these studies in emphasis: rather than building a new detector or segmenter, it studies how to accelerate an existing full workflow without breaking downstream engineering outputs.

\hypertarget{why-the-current-problem-is-non-trivial}{%
\section{Why the Current Problem Is Non-Trivial}\label{why-the-current-problem-is-non-trivial}}

The current workflow is harder than a standard benchmark for three reasons: (i)~the pipeline is jointly model-bound and systems-bound, with detection preparation, artifact management, and reporting all capable of dominating wall-clock time; (ii)~downstream metrology applies thresholded geometric logic that can change abruptly from a few-pixel boundary shift; and (iii)~full-case Dice and patch latency can mask deployment risk, since a method can appear strong in isolation yet fail after whole-pipeline integration.

\hypertarget{system-overview}{%
\section{System Overview}\label{system-overview}}

The acceleration framework developed in this report is built around two
connected repositories. \emph{wp5-seg} is responsible for model
design, pruning, finetuning, evaluation, and deployment preparation.
\emph{intelliscan} is responsible for end-to-end raw-scan inference.
The practical design principle is that the segmentation model cannot be
optimized in isolation. Every candidate model or runtime shortcut must
eventually be validated inside the full \emph{intelliscan} workflow.

The overall processing chain is:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Load a raw 3D X-ray scan.
\item
  Generate two-view 2D slices for detection.
\item
  Detect candidate regions using YOLO-style 2D detection {[}12,13{]}.
\item
  Merge detections into 3D bounding boxes.
\item
  Segment each bump region with a 3D network.
\item
  Compute metrology and defect flags from the predicted components.
\item
  Generate human-readable artifacts such as report pages and optional
  GLTF outputs.
\end{enumerate}

This decomposition made it possible to optimize not only the
segmentation model, but also the data movement and orchestration around
it.

\begin{figure}
\centering
\includegraphics[width=0.82\textwidth]{../figures/pipeline_overview_demo.png}
\caption{End-to-end artifact flow of the current \emph{intelliscan}
default pipeline.}
\end{figure}

As a concrete example, processing SN002 (95~detected bumps) through the full chain produces: 3D bounding boxes, an assembled segmentation volume, a per-bump metrology CSV, and a human-readable PDF report---all from a single raw NIfTI input.

\subsection{Deliverable Inspection Report}

The final output of the pipeline is a per-scan inspection report designed for engineering review. For each detected bump, the metrology stage computes the following measurements (all in $\mu$m at 0.7\,$\mu$m voxel resolution):

\begin{itemize}
\tightlist
\item \textbf{Bond-Line Thickness (BLT):} vertical distance between the copper pillar bottom and the solder top surface.
\item \textbf{Pad Misalignment:} Euclidean offset between the copper-pillar centroid and the copper-pad centroid.
\item \textbf{Void-to-Solder Ratio:} volumetric fraction of void voxels within the solder region; the primary defect indicator.
\item \textbf{Solder Extrusion:} lateral overflow of solder beyond the pillar boundary, measured on four sides (left, right, front, back).
\item \textbf{Pillar Dimensions:} width and height of the copper pillar.
\end{itemize}

Each measurement is compared against a threshold to produce a binary defect flag:

\begin{table}[H]
\centering
\caption{Defect flag thresholds in the delivered inspection report}
\label{tab:defect-thresholds}
\renewcommand{\arraystretch}{1.15}
\begin{tabular}{l l l}
\toprule
\textbf{Defect Flag} & \textbf{Condition} & \textbf{Threshold} \\
\midrule
Void-ratio defect       & void/solder ratio $>$ threshold & 30\% \\
Pad-misalignment defect & misalignment / pillar width $>$ threshold & 10\% \\
Solder-extrusion defect & any-side extrusion / pillar width $>$ threshold & 10\% \\
\bottomrule
\end{tabular}
\end{table}

The deliverable CSV contains one row per bump with all measurements and defect flags. The PDF report presents a visual summary with segmentation overlays, measurement annotations, and a defect summary table. This structure enables engineers to review flagged bumps without re-running the analysis. The acceleration work in this report preserves the content and format of these deliverables exactly---the safe engineering path produces byte-identical CSV and PDF outputs.

\hypertarget{model-level-design}{%
\section{Model-Level Design}\label{model-level-design}}

Figure~\ref{fig:kd-pruning-arch} provides an overview of the model-level acceleration pipeline. The heavy teacher (BasicUNet 3D, features $32 \to 32 \to 64 \to 128 \to 256 \to 32$) is first compressed via structured channel pruning at a configurable ratio~$r$. The pruned student is then fine-tuned under a combined loss that includes standard supervision ($L_\text{sup}$) and a class-aware knowledge distillation term ($L_\text{KD}$) operating at the logit level. Class affinity weights $\alpha_c$ up-weight the defect-relevant classes (void and pad) to preserve the boundaries that feed downstream metrology. The following subsections detail each component.

\centering

\caption{Class-aware knowledge distillation and structured pruning pipeline. The teacher's logits guide the pruned student through a masked KL loss with per-class affinity weights $\alpha_c = [1,1,1,2,2]$, combined with standard supervision loss.}

\hypertarget{structured-pruning}{%
\subsection{Structured Pruning}\label{structured-pruning}}

The report explores structured pruning ratios rather than only a single
pruned model. Screening experiments established that pruning ratio
0.25 was not worth continuing, while 0.375,
0.5, and 0.625 formed a useful first Pareto set. This
design supports a more convincing report because it shows a
latency-quality frontier instead of one arbitrary compressed checkpoint,
which is also more aligned with deployment-oriented channel-pruning
literature {[}4,14,15{]}.

For a layer \(l\) with original channel count \(C_l\), the simplified
pruning rule used in this report can be written as

\[
C'_l = \left\lfloor (1-r) C_l \right\rfloor
\]

where \(r \in [0,1)\) is the pruning ratio and \(C'_l\) is the retained
channel count after structured channel removal. In practice, the
retained widths are also constrained to remain valid BasicUNet feature
sizes for the pruned checkpoint and deployment scripts, but the equation captures the central design variable used throughout the ratio sweep.

The channels to remove are selected by importance scoring. For each output channel $j$ in a convolutional layer followed by InstanceNorm, the importance is computed as:
\[
s_j = \| \mathbf{W}_j \|_2 \cdot |\gamma_j|
\]
where $\| \mathbf{W}_j \|_2$ is the L2-norm of the weight tensor for channel $j$ and $\gamma_j$ is the corresponding InstanceNorm scale parameter {[}4,14{]}. Channels with the lowest $s_j$ are removed. To respect the U-Net skip connections, encoder and decoder levels are pruned symmetrically so that concatenation dimensions remain valid.

\hypertarget{knowledge-distillation}{%
\subsection{Knowledge Distillation}\label{knowledge-distillation}}

The retained distillation design is immediate class-aware logit
distillation {[}5{]} with:

\begin{itemize}
\tightlist
\item
  distillation weight 0.1
\item
  temperature 2.0
\item
  class weights 1,1,1,2,2
\end{itemize}

The supervised term and the retained class-aware distillation term are
implemented as

\[
L_{\mathrm{sup}} = 0.5 L_{\mathrm{CE}} + 0.5 L_{\mathrm{Dice}}
\]

\[
L_{\mathrm{KD}} =
\frac{T^2}{\sum_i m_i \alpha_{y_i}}
\sum_i m_i \alpha_{y_i}
\operatorname{KL}
\left(
\operatorname{softmax}\left(\frac{z_t^{(i)}}{T}\right)
\;\middle\|\;
\operatorname{softmax}\left(\frac{z_s^{(i)}}{T}\right)
\right)
\]

\[
L_{\mathrm{total}} = L_{\mathrm{sup}} + \lambda_{\mathrm{KD}} L_{\mathrm{KD}}
\]

where \(z_t^{(i)}\) and \(z_s^{(i)}\) are teacher and student logits at
voxel \(i\), \(m_i\) is the valid-voxel mask that removes the ignore
label, \(\alpha = [1,1,1,2,2]\) is the retained class-weight vector,
\(T = 2.0\), and \(\lambda_{\mathrm{KD}} = 0.1\). This formulation
matches the implemented masked KL loss used in the retained full-case
student.

This configuration was not chosen by intuition alone. Several
alternatives were tested and rejected:

\begin{itemize}
\tightlist
\item
  delayed KD did not outperform immediate KD,
\item
  class-aware weight 1,1,1,4,2 did not remain best after
  full-case validation,
\item
  feature-level attention KD did not beat the retained logit-KD
  baseline,
\item
  supervised class reweighting and boundary-loss additions did not
  survive full-case or downstream validation.
\end{itemize}

The design implication is important: for this task, a lightweight
class-aware KD term was more useful than more complex feature or
auxiliary-loss variants under the tested conditions.

\hypertarget{full-case-and-deployment-aware-selection}{%
\subsection{Full-Case and Deployment-Aware
Selection}\label{full-case-and-deployment-aware-selection}}

The report adds a two-stage model-selection rule. A compressed
checkpoint is not kept merely because it improves patch latency. It must
also:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  preserve good full-case Dice on the 174-case set, and
\item
  preserve acceptable behavior after deployment into
  \emph{intelliscan}.
\end{enumerate}

This led to an additional design refinement: checkpoint selection inside
the same training run. Instead of using only \emph{best.ckpt} by
full-case Dice, the report introduced a checkpoint panel through the
\emph{--save-eval-checkpoints} helper and tested whether
intermediate epochs behave differently in the downstream pipeline.

The old pipeline segmented each bbox, saved per-bbox prediction NIfTIs,

preserving output compatibility. This design keeps segmentation geometry
unchanged and only reduces redundant artifact flow.

\hypertarget{report-reconstruction-without-per-bbox-raw-crops}{%
\subsection{Report Reconstruction Without Per-BBox Raw

The report generation stage was redesigned to reconstruct visualization

\hypertarget{gltf-decoupling-from-per-bbox-prediction-niftis}{%
\subsection{GLTF Decoupling from Per-BBox Prediction
NIfTIs}\label{gltf-decoupling-from-per-bbox-prediction-niftis}}

After report reconstruction stopped needing \texttt{mmt/img}, GLTF
export remained the last reason to keep per-bbox prediction files in the
default combined path. The report therefore moved GLTF generation to an
in-memory interface that consumes the already available segmentation
results. This further reduced artifact clutter and stage-specific
overhead.

\hypertarget{exact-in-memory-detection}{%
\subsection{Exact In-Memory Detection}\label{exact-in-memory-detection}}

The detection path underwent several iterations. Simply moving images
into memory was not sufficient to preserve parity because the legacy
path implicitly quantized detections through \texttt{.txt} serialization
before 3D merge. The retained design therefore keeps the no-write
detection path, but explicitly reproduces the legacy two-decimal
rounding behavior before bbox merge. That design change was necessary to
make in-memory detection both fast and exact.

The retained parity-preserving logic is summarized below.

\begin{pseudocode}[Algorithm 1: Exact In-Memory Detection with Legacy-Equivalent Quantization]
\textbf{Input:} raw 3D volume $V$, YOLO detector $\mathcal{M}$ \quad \textbf{Output:} merged 3D bounding boxes $\mathbf{B}_\text{3D}$ \\
\textbf{for each} detection view $v \in \{v_1, v_2\}$ \textbf{do} \\
\quad Slice $V$ into ordered 2D images along view $v$ \\
\quad Apply identical PIL transform as the legacy file-based path \\
\quad Perform in-memory JPEG encode/decode roundtrip \hfill\textit{$\triangleright$ preserve quantization} \\
\quad $\mathbf{D}_v \gets \mathcal{M}(\text{decoded images})$ \hfill\textit{$\triangleright$ YOLO inference} \\
\quad \textbf{for each} detection $d \in \mathbf{D}_v$ \textbf{do} \\
\quad\quad $d.\text{class\_id} \gets \lfloor d.\text{class\_id} \rceil$ \hfill\textit{$\triangleright$ round to integer} \\
\quad\quad $d.\text{coords} \gets \mathrm{round}(d.\text{coords},\, 2)$ \hfill\textit{$\triangleright$ two-decimal quantization} \\
\quad \textbf{end for} \\
\textbf{end for} \\
$\mathbf{B}_\text{3D} \gets \textsc{Merge}(\mathbf{D}_{v_1},\, \mathbf{D}_{v_2})$ \\
\textbf{return} $\mathbf{B}_\text{3D}$ \\
\end{pseudocode}

The key engineering insight is that exact parity did not come from
``using memory instead of disk'' alone. It came from reproducing the
legacy numeric behavior---specifically the two-decimal coordinate
quantization---at the point where 2D detections are converted into
the coordinates consumed by the 3D merge stage.

\hypertarget{cpu-preparation-parallelism-and-batch-prefetch}{%
\subsection{CPU Preparation Parallelism and Batch
Prefetch}\label{cpu-preparation-parallelism-and-batch-prefetch}}

Once detection became exact in-memory, the next bottleneck was CPU-side
slice preparation. The report therefore parallelized only the per-slice
CPU image-preparation step and then overlapped next-batch preparation
with current YOLO inference. These optimizations were intentionally
narrow so that they changed scheduling without changing numerical
behavior.

\hypertarget{algorithmic-design-and-causal-ablation}{%
\section{Algorithmic Design and Causal
Ablation}\label{algorithmic-design-and-causal-ablation}}

\hypertarget{adaptive-margin}{%
\subsection{Adaptive Margin}\label{adaptive-margin}}

Adaptive margin was originally investigated to activate more direct-path
segmentation by shrinking expanded crops into the ROI. However, the
report did not stop at timing results. It added factor-isolation
controls to separate:

\begin{itemize}
\tightlist
\item
  crop/context change,
\item
  inference-path change, and
\item
  downstream metrology sensitivity.
\end{itemize}

This led to a clear causal finding: on clipped cases such as
SN009, the main source of drift was context shrinkage, not the
direct-padded fast path itself.

\hypertarget{guarded-adaptive-margin}{%
\subsection{Guarded Adaptive Margin}\label{guarded-adaptive-margin}}

Because full adaptive margin was unsafe, the report tested guarded
candidates that only allowed limited context loss. This search showed
that a narrow region such as $xy \leq 12$,
$z \leq 24$ could recover part of the speedup while
reducing most of the drift. Even so, the best guarded candidate still
produced a non-zero defect flip, so it was kept only as an experimental
result rather than a default method.

\hypertarget{larger-direct-padded-roi}{%
\subsection{Larger Direct Padded ROI}\label{larger-direct-padded-roi}}

The report also tested a geometry-preserving alternative: keep the real
crop unchanged, but enlarge the direct padded shell. This idea turned
out to be unsafe. The experiments showed that even when the crop
geometry and routing stayed almost unchanged, the larger padded input
canvas could still alter predictions. This finding is consequential because
it shows that preserving bbox extent alone is not enough; the numerical
behavior of the direct path also depends on the padded canvas size.

\hypertarget{two-level-deployment-safety-gate}{%
\section{Two-Level Deployment-Safety
Gate}\label{two-level-deployment-safety-gate}}

The central evaluation methodology of this report is the \textbf{Two-Level Deployment-Safety Gate}, which formalises the principle that downstream-aware ranking must replace Dice-only ranking in threshold-sensitive applications. The gate operates as follows: a candidate acceleration must pass both levels sequentially; failure at either level excludes the method from the deployment-safe set.

Table~\ref{tab:gate-criteria} defines the operationalised pass criteria for each gate. These thresholds are informed by the metrology decision boundaries in the current pipeline (Section~1.1.1) and by the practical requirement that a deployed model must not introduce defect-flag disagreements relative to the validated teacher.

\begin{table}[H]
\centering
\caption{Operationalised pass criteria for the Two-Level Deployment-Safety Gate}
\label{tab:gate-criteria}
\renewcommand{\arraystretch}{1.2}
\begin{adjustbox}{max width=\textwidth}
\begin{tabular}{l l l p{0.38\textwidth}}
\toprule
\textbf{Gate} & \textbf{Metric} & \textbf{Pass Condition} & \textbf{Rationale} \\
\midrule
\multirow{3}{*}{Gate 1 (\emph{wp5-seg})}
  & Full-case avg Dice  & $\geq$ 0.88  & Ensures overall segmentation quality is not substantially degraded \\
  & Class~3 (void) Dice & $\geq$ 0.70  & Void is the primary defect class; large drops signal lost sensitivity \\
  & Class~4 (pad) Dice  & $\geq$ 0.85  & Pad geometry directly affects misalignment measurement \\
\midrule
\multirow{4}{*}{Gate 2 (\emph{intelliscan})}
  & Pad-misalignment flips   & $= 0$  & Any flip is a false alarm or missed defect in production \\
  & Solder-extrusion flips   & $= 0$  & Same rationale as pad flips \\
  & Mean BLT $|\Delta|$      & $< 1.0\,\mu$m & Keeps geometric measurement within one pixel tolerance \\
  & WP class~3 / class~4 Dice & $\geq$ 0.90 & Ensures per-scan segmentation stability, not just dataset-level \\
\bottomrule
\end{tabular}
\end{adjustbox}
\end{table}

In practice, the zero-flip requirement at Gate~2 is the strictest criterion and is the one most frequently violated by compressed models that pass Gate~1 comfortably. This operationalisation transforms the evaluation from an informal judgement into a reproducible pass/fail protocol.

Two clarifications on these thresholds are necessary. First, the Gate~1 values (avg Dice $\geq$ 0.88, class~3 $\geq$ 0.70, class~4 $\geq$ 0.85) are conservative screening thresholds informed by the observed teacher performance, not theoretically optimal cutoffs; class~3 is set lower because void is the rarest and hardest class, while class~4 is set higher because pad geometry directly feeds the misalignment measurement. The BLT bound of 1.0\,$\mu$m corresponds to one pixel at the current 0.7\,$\mu$m voxel resolution, providing a single-pixel tolerance. Second, Gate~2 adopts zero defect-flag flips because defect flags are the final production decision variable and therefore require stricter tolerance than Dice. The WP Dice entries at Gate~2 refer to \emph{teacher-relative} whole-pipeline Dice, since raw-scan voxel-level ground truth is unavailable at this stage; the teacher output serves as the reference standard.

\hypertarget{experimental-setup-and-metrics}{%
\section{Experimental Setup and
Metrics}\label{experimental-setup-and-metrics}}

The experiments in this report were carried out on local NVIDIA L40 GPUs
using the \emph{intelliscan} and \emph{wp5-seg} environments in the
current workspace. Two groups of data were used.

For model-level evaluation, the compressed segmentation models were
tested on the full 174-case evaluation set used by
\emph{wp5-seg}.

For pipeline-level evaluation, raw scans from SN002,
SN003, SN008, SN009, SN010,
SN011, SN012, and SN061 were used, depending
on the experiment.

The most important metrics in this report are:

\begin{itemize}
\tightlist
\item
  average Dice on the full segmentation test set,
\item
  class~3 (void) and class~4 (pad) Dice as primary quality indicators,
\item
  end-to-end runtime,
\item
  stage runtime,
\item
  defect-flag flips,
\item
  BLT and pad-misalignment deltas.
\end{itemize}

Although all five segmentation classes are evaluated, class~3 (void) and class~4 (copper pad) receive particular emphasis throughout this report for two reasons. First, void is the primary defect indicator: void ratio directly determines the void-defect flag, so class~3 Dice is the most safety-critical segmentation metric. Second, pad geometry feeds directly into the pad-misalignment measurement (see Equation~1.1); inaccurate pad boundaries can flip the misalignment flag even when overall Dice appears acceptable. Classes~0--2 (background, copper pillar, solder) are substantially easier to segment and rarely change across acceleration variants, so they provide little discriminative power for comparing methods. Full per-class results are reported in the master table and the Pareto ranking table for completeness.

This report intentionally does not rely on voxel agreement alone when
judging acceptability.

\hypertarget{case-study-good-full-case-dice-bad-pipeline-behavior}{%
\subsection{Case Study: Good Full-Case Dice, Bad Pipeline
Behavior}\label{case-study-good-full-case-dice-bad-pipeline-behavior}}

The $r$\,=\,0.5 student provides the clearest deployment-gap example.
At model level, it looks strong: its full-case average Dice is
0.889527, with class 3 = 0.754505 and class
4 = 0.874937 on the full 174-case evaluation set.
However, once this student is inserted into the whole raw-scan pipeline,
the downstream picture changes sharply.

On SN002, the $r$\,=\,0.5 eager deployment produces
65/95 solder-extrusion defect flips relative to the
teacher. On SN009, the same student produces
63/114 pad-misalignment defect flips. The backend does not
rescue this behavior: the compile and TRT FP16 rows remain numerically
near-identical to the eager row on the tested panel. This case study demonstrates the main methodological claim of the report: high full-case Dice is necessary, but not sufficient, for deployment into a thresholded metrology workflow.

The underlying mechanism can be understood as follows. Structured pruning and low-precision quantization both reduce the model's capacity to represent high-frequency spatial features, particularly at class boundaries. Since the interior of large regions (background, pillar, solder bulk) is easy to predict correctly, overall Dice remains high even when boundary voxels shift by a few pixels. However, the metrology stage computes geometric quantities (pad offset, solder extrusion width) directly from these boundaries: a shift of 2--3 voxels at the pad--solder interface can change the measured misalignment ratio by several percent, crossing the 10\% defect threshold. This explains why the deployment gap is concentrated at class~3 (void) and class~4 (pad)---the two classes whose boundaries feed the most threshold-sensitive measurements---while classes~0--2 remain virtually unchanged.

\subsection{Deployment Recommendation}

Based on the full evidence in this report, three deployment tiers can be identified:

\begin{enumerate}
\item \textbf{Recommended for immediate deployment: Row~4 (safe engineering path).} This configuration uses the original teacher model with all engineering optimizations enabled. It achieves \textbf{1.94$\times$ end-to-end speedup} (93.23\,s $\to$ 48.15\,s) with \emph{zero quality loss}---identical segmentation, identical metrology, identical defect flags. No model change is required; only the pipeline orchestration is optimized. This is the strongest deployable result in this report.

\item \textbf{Conditionally deployable: Row~6 (Teacher / TRT INT8 with real-data calibration).} This adds TRT INT8 backend acceleration to reach \textbf{2.38$\times$ speedup}. Segmentation quality remains high (WP Dice 0.948/0.957), solder flips are zero, and BLT drift is minimal (0.25\,$\mu$m). However, 7~pad-misalignment flips out of 114~bumps (6.1\%) prevent full Gate~2 clearance. If the application can tolerate a small pad-flip rate, or if the remaining flips can be resolved through targeted calibration refinement, this configuration offers meaningful additional speedup at near-production quality.

\item \textbf{Not recommended for deployment: Rows~7--8 (compressed students).} Despite achieving up to \textbf{3.17$\times$ speedup} and strong full-case Dice ($\geq$ 0.889), the compressed students produce unacceptable defect-flag instability (52--141 flips) upon whole-pipeline deployment. The \textbf{10.7$\times$ patch-level speedup} ($r$\,=\,0.5 / TRT FP16: 1.62\,ms vs.\ teacher 17.40\,ms) confirms that model compression is highly effective at the kernel level, but whole-pipeline gains are limited by non-model bottlenecks, while downstream quality cost remains prohibitive.
\end{enumerate}

The practical implication is that \textbf{the teacher model with engineering optimization (Row~4) should be the default deployment target}. Model compression should only be revisited after the deployment gap is closed through downstream-aware training strategies.

\hypertarget{evidence-limitations}{%
\section{Evidence Limitations}\label{evidence-limitations}}

Three limitations of the current evidence base should be noted explicitly. First, the raw-scan evaluation panel contains at most eight samples; while this is sufficient to demonstrate the deployment gap, a larger panel would strengthen the statistical confidence of Gate~2 verdicts. Second, although Section~4.1.1 confirms low runtime variance for the TRT~FP16 backend ($\pm$\,0.31\,s), some whole-pipeline quality comparisons still rely on single sequential runs per sample; extending these to repeated runs with confidence intervals remains desirable. Third, Gate~2 uses teacher-relative whole-pipeline segmentation as the reference standard because voxel-level ground truth is unavailable for raw scans; if ground-truth annotations become available in future, the gate criteria should be re-validated against them.

\hypertarget{conclusions}{%
\chapter{Conclusions}\label{conclusions}}

\hypertarget{what-has-been-done}{%
\section{What Has Been Done}\label{what-has-been-done}}

In summary, this report establishes a generalizable principle for
threshold-sensitive industrial pipelines: \emph{any acceleration method
must be validated through both model-level segmentation quality and
whole-pipeline defect stability before it can be considered
deployment-safe.} Applying this principle to a practical 3D
semiconductor inspection workflow, the report achieves large engineering
speedups while demonstrating why several superficially attractive
shortcuts fail the deployment-safety criterion. The
study covered model-level compression, algorithmic ablation, runtime
behavior, and pipeline engineering. The work did not stop at patch-level
speed or segmentation overlap. Instead, it evaluated every important
decision against the whole \emph{intelliscan} workflow whenever
possible.

At the engineering level, the report validated a sequence of safe
optimizations that significantly reduce runtime while preserving exact
outputs on the local validated raw-scan set. The strongest result is the
final safe default front-end path, which combines exact quantized
in-memory detection, CPU image-preparation parallelism, and intra-view
batch prefetch. Relative to the older file-based detection baseline,
this path reduces mean total runtime by 45.13\% and pooled
total runtime by 44.20\% on eight validated raw scans.

At the model level, the report established a first pruning Pareto front
and identified a retained class-aware KD recipe that improves
compressed-student recovery over plain finetuning on the full
174-case evaluation set. At runtime level, the report completed
a retained backend ranking for the current compressed students.
$r$\,=\,0.5 / TRT FP16 is the strongest retained speed-and-energy
runtime point, while $r$\,=\,0.375 / TRT FP16 is the stronger
quality-oriented retained runtime point. At the algorithmic level, the
report also produced causal evidence explaining why some faster
segmentation policies are unsafe. In particular, adaptive-margin drift
was shown to be dominated by context shrinkage, while larger direct
padded ROI experiments showed that the padded input canvas itself can
change predictions.

The report also documented a critical deployment lesson: good full-case
Dice does not guarantee good downstream metrology. This is one of the
most important outcomes of the work and directly supports the
whole-pipeline-first evaluation philosophy adopted in the project.

\hypertarget{what-has-not-been-done}{%
\section{What Has Not Been Done}\label{what-has-not-been-done}}

Several important items remain unfinished.

First, no compressed student has yet cleared the full whole-pipeline
deployment gate strongly enough to replace the teacher model in
\emph{intelliscan}. Even the best current checkpoint-panel candidate
improves some downstream metrics while remaining mixed on others.

Second, no compressed student plus runtime backend pair is yet
deployment-safe enough to replace the teacher inside the current
metrology-sensitive workflow. Backend ranking is now complete enough for
the present students, but the dominant drift source remains the student
model rather than the backend choice.

Third, \emph{torch.compile(reduce-overhead)} is useful as a PyTorch
fallback backend in \emph{wp5-seg}, but it is not a good current
single-scan \emph{intelliscan} backend because cold-start compile time
overwhelms the steady-state patch-level gain in the present CLI
workflow.

Fourth, real-data INT8 calibration and QAT have both been completed for the teacher model. PTQ INT8 comes closest to Gate~2 compliance (WP Dice 0.948/0.957, zero solder flips, BLT $\Delta$ 0.25\,$\mu$m), but 7~pad-misalignment flips still prevent full clearance. A 10-epoch QAT attempt did not improve upon PTQ, indicating that closing the remaining gap requires longer training or boundary-aware QAT strategies.

Fifth, actual GLTF output generation still needs one final recheck in a
PyVista-enabled environment, although the artifact-dependency cleanup
itself is already validated.

\hypertarget{extensions-and-future-work}{%
\section{Extensions and Future Work}\label{extensions-and-future-work}}

If more time were available, the next extensions should follow the same
defect-aware logic rather than a speed-first logic.

The first extension should be downstream-aware model selection or
training. The current checkpoint-panel study already shows that
checkpoint ranking changes after deployment. A stronger next step would
be to incorporate a small raw-scan validation panel directly into
checkpoint selection or early stopping.

The second extension should be real-data TensorRT deployment of the best
retained student. This should begin with FP16 and then move to PTQ or
QAT only after real calibration data are prepared. However, no
deployment claim should be made unless class-sensitive and
metrology-sensitive stability are preserved.

The third extension should be a stronger geometry-preserving runtime
design. The negative results in this report have already shown which
directions are not promising. Future work should therefore avoid context
trimming and avoid naive larger-padding shells unless the direct path is
retrained or explicitly aligned.

Finally, this report can be extended into a publication-quality study by
turning the current validated tables into a unified
latency-versus-quality framework, where model size, average Dice, class
3, class 4, defect flips, and metrology deltas are
reported together rather than in isolation.
