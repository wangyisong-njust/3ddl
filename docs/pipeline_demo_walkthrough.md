# Pipeline Demo Walkthrough

## Goal

This note gives a concrete end-to-end example of what goes into the current `intelliscan` pipeline and what comes out of it. It is intended as thesis/report-ready material for explaining the full artifact flow on one real scan.

## Representative Demo Case

- Input raw scan:
  - one representative `SN002` raw NIfTI volume from the local raw-scan validation set
- Output bundle from the current safe default path:
  - the corresponding `SN002` output directory generated by `intelliscan`

The run used the current validated safe default front-end:

- exact quantized in-memory detection
- `4` CPU image workers
- intra-view batch prefetch

## What Goes In

The pipeline input is a single 3D X-ray reconstruction stored as a NIfTI volume (`*.nii`). The user-facing invocation is simply:

```bash
cd intelliscan
python main.py /path/to/input.nii --force
```

At this stage, the system has not yet created any detections, segmentations, or measurements. The only required external assets are the detection and segmentation model weights configured for the repository.

## What Happens Internally

For the representative `SN002` run, the pipeline produced the following stages:

| Stage | Time (s) |
| --- | ---: |
| Folder Creation | `0.0001` |
| 2D Detection Inference | `9.5277` |
| 3D Bounding Box Generation | `0.4441` |
| 3D Segmentation + Metrology | `14.1753` |
| GLTF Generation | `0.0004` |
| Report Generation | `0.8214` |
| Total | `24.9691` |

The important intermediate and final artifacts are:

- `bb3d.npy`: merged 3D bump boxes (`95` boxes in this run)
- `segmentation.nii.gz`: full 3D segmentation volume
- `metrology/metrology.csv`: one row per bump with BLT, pad misalignment, solder extrusion, and defect flags
- `metrology/final_report.pdf`: human-readable inspection report
- `metrics.json`: phase timing log for profiling

## What Comes Out

For this `SN002` demo:

- the pipeline generated `95` bump-level metrology rows
- the representative bump shown in Fig. 3.2 had:
  - `BLT = 33.6`
  - `Pad_misalignment = 14.85`
  - `solder_extrusion_defect = True`
  - `pad_misalignment_defect = True`
- the final PDF report was written to:
  - the `metrology/final_report.pdf` artifact inside the corresponding `SN002` output bundle

This example is useful in the thesis because it makes the pipeline outputs concrete. The system does not stop at segmentation. It produces both machine-readable outputs (`bb3d.npy`, `segmentation.nii.gz`, `metrology.csv`, `metrics.json`) and human-readable deliverables (`final_report.pdf` and optional GLTF files).

## Why This Demo Matters

This walkthrough supports the main thesis claim of the project: the real deployment target is the full inspection workflow, not the segmentation model in isolation. A method that improves patch latency or even full-case Dice must still preserve the downstream bump-level measurements and defect decisions generated by the whole pipeline.
